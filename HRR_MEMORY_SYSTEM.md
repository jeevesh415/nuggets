## 1. What This System Actually Is

This is a **Holographic Reduced Representation (HRR)** memory that encodes structured facts as superposed complex-valued tensors and retrieves them algebraically. It sits in front of an LLM (Gemini) as a **fast, constant-size cache** that:

- Answers queries in microseconds without API calls when it has the answer
- Falls back to the LLM when uncertain
- **Learns from its own fallbacks** by upserting the LLM's answer into holographic memory
- Over time, absorbs the LLM's knowledge into a single tensor

It is not a vector database. It is not RAG. It is a **distributed associative memory** where every fact lives superimposed in the same mathematical object.

---

## 2. The "Sticky Notes" Mental Model

**Yes, this system is like sticky notes for an LLM — but with a crucial twist.**

### How it's like sticky notes:
- Each fact is a small, self-contained key-value pair (like writing "WiFi Password: test123" on a sticky note)
- Facts are extracted from a source document and kept for quick reference
- The system checks its notes before bothering to re-read the full document
- New notes can be added at any time

### How it's MORE than sticky notes:
- **All notes are compressed into a single object.** Imagine taking 50 sticky notes and holographically projecting them all onto the same piece of film. The film is the same size regardless of whether you stored 5 notes or 50. That's what superposition does.
- **Notes are retrieved by algebraic query, not scanning.** You don't look through sticky notes one by one. You multiply the hologram by the conjugate of your query key, and the relevant note's content emerges from the interference pattern.
- **Notes interfere with each other.** Unlike real sticky notes, adding more facts degrades the signal quality of all retrievals. The system manages this with sharding, ensembles, and signal processing.
- **Notes can be "fuzzy-matched."** The cosine similarity decoding means you can ask for something *close* to a stored key and still get the right answer.

### The real analogy:
**Sticky notes** = discrete, separate, individually addressable, no interference, unlimited capacity, O(N) scan time.
**HRR memory** = continuous, superposed, algebraically addressable, controlled interference, capacity ~√D, O(1) retrieval.

A better analogy than sticky notes: **it's like a brain's working memory** — a fixed-capacity space where recently learned facts are held in a compressed, overlapping representation, queryable by association rather than by address.

### Compared to existing systems:
- **RAG** = a filing cabinet (separate documents, search by similarity, return exact text)
- **OpenClaw memory** = a consciously managed journal (agent writes what it wants to remember, searches by hybrid vector+keyword)
- **HRR memory** = a holographic sticky-note pad (all notes superposed into one object, retrieved by algebraic pattern-matching)

## 3. Mathematical Foundations

### 3.1 Complex Unit-Magnitude Keys

Every entity in the system (vocabulary words, sentence IDs, role positions) is represented as a **unit-magnitude complex vector** of dimension D:

```
key ∈ ℂ^D,  where |key_i| = 1 for all i
```

Generated by sampling random phases:

```
φ ~ Uniform(0, 2π)^D
key = e^(iφ) = cos(φ) + i·sin(φ)
```

This is **FHRR (Fourier HRR)** — all keys live on the unit circle in the complex plane. The critical property: the conjugate of a key is its exact inverse under element-wise multiplication.

### 3.2 Binding (Multiplicative)

Binding two keys produces a new key that is **dissimilar to both inputs**:

```
bind(a, b) = a ⊙ b     (element-wise complex multiplication)
```

Properties:
- **Commutative**: `bind(a,b) = bind(b,a)`
- **Invertible**: `bind(bind(a,b), conj(b)) = a` (exactly, for unit-magnitude keys)
- **Distributes over addition**: `bind(a, b+c) = bind(a,b) + bind(a,c)`
- **Preserves dimensionality**: input D → output D

### 3.3 Unbinding (Conjugate Multiplication)

To retrieve `a` from `bind(a,b)`, multiply by `conj(b)`:

```
unbind(memory, key) = memory ⊙ conj(key)
```

For a single binding, this is exact. For a superposition of bindings, this produces `a + noise` where noise comes from all other superposed items.

### 3.4 Superposition (Additive)

Multiple bindings are summed into a single memory vector:

```
memory = Σ_i  bind(S_i, bind(R_i, W_i))
```

With optional scaling to control magnitude:

```
memory = memory × (1 / √N)    where N = number of bindings
```

### 3.5 Three-Way Binding Structure

Each fact is encoded as a triple binding:

```
fact_binding = Sentence_key ⊗ Role_key ⊗ Word_key
```

Where:
- **Sentence_key**: identifies WHICH fact (e.g., fact #3)
- **Role_key**: identifies the POSITION within the fact (e.g., position 0)
- **Word_key**: identifies the VALUE (e.g., "test123")

This gives structured, addressable memory: "what value is at position P of sentence S?" is computed as:

```
retrieved = unbind(unbind(memory, S), R)
decoded = argmax_w cosine_similarity(retrieved, vocab[w])
```

### 3.6 Role Keys (Power Encoding)

Role keys use a **power-based scheme** rather than random keys:

```
base = random unit-magnitude complex vector of dim D
role[k] = base^k    (element-wise exponentiation)
```

This creates a natural ordering: position 0, 1, 2, ... are related by successive multiplication, which preserves sequential structure.

### 3.7 Cosine Similarity (in ℝ^{2D})

For decoding, complex vectors are converted to real vectors by concatenating real and imaginary parts:

```
real_vec(z) = [Re(z), Im(z)] ∈ ℝ^{2D}
```

Then unit-normalized and compared via dot product:

```
similarity(a, b) = real_vec(a)·real_vec(b) / (‖real_vec(a)‖ · ‖real_vec(b)‖)
```

### 3.8 Vocabulary Orthogonalization

To reduce cross-interference between vocabulary items, a Gram-Schmidt-like decorrelation is applied:

```
K ∈ ℝ^{V×2D}   (vocab keys, real representation)
For each iteration:
    G = K @ K^T           (Gram matrix)
    G.diagonal = 0        (ignore self-similarity)
    K = K - step × (G @ K) / 2D    (subtract correlated components)
    K = K / ‖K‖           (re-normalize rows)
Convert back to unit-magnitude complex via atan2
```

This makes vocabulary items more orthogonal, improving retrieval discrimination.

## 4. Key Algorithms (Exact Code)

### 4.1 Key Generation

```python
@torch.no_grad()
def make_vocab_keys_complex(V: int, D: int, device, dtype) -> torch.Tensor:
    phi = 2 * math.pi * torch.rand(V, D, device=device, dtype=dtype)
    return torch.polar(torch.ones_like(phi), phi)
```

### 4.2 Binding & Unbinding

```python
class ComplexBinder:
    def __init__(self, dim: int, device, dtype):
        self.dim, self.device, self.dtype = dim, device, dtype

    @torch.no_grad()
    def bind(self, a: torch.Tensor, b: torch.Tensor) -> torch.Tensor:
        return a * b

    @torch.no_grad()
    def unbind(self, m: torch.Tensor, key: torch.Tensor) -> torch.Tensor:
        return m * torch.conj(key)
```

### 4.3 Orthogonalization

```python
@torch.no_grad()
def orthogonalize_complex(keys: torch.Tensor, iters: int = 1, step: float = 0.4) -> torch.Tensor:
    if iters <= 0: return keys
    K = torch.cat([keys.real, keys.imag], dim=-1)
    D2 = K.shape[-1]
    for _ in range(iters):
        G = K @ K.T
        G.fill_diagonal_(0.0)
        K = K - (step * (G @ K) / D2)
        K = K / (K.norm(dim=-1, keepdim=True) + 1e-9)
    real, imag = torch.tensor_split(K, 2, dim=-1)
    phase = torch.atan2(imag, real)
    return torch.polar(torch.ones_like(phase), phase)
```

### 4.4 Signal Processing

```python
@torch.no_grad()
def sharpen_complex(z: torch.Tensor, p: float = 1.0, eps: float = 1e-12):
    """p > 1 => contrast-increasing magnitude nonlinearity"""
    if p == 1.0: return z
    mag = torch.abs(z)
    return z * (mag + eps) ** (p - 1.0)

@torch.no_grad()
def corvacs_lite(z: torch.Tensor, a: float = 0.0):
    """Gentle magnitude limiter (a=0 disables)."""
    if a <= 0: return z
    mag = torch.abs(z) + 1e-12
    return z * (torch.tanh(a * mag) / mag)

def softmax_temp(sims: torch.Tensor, T: float = 1.0) -> torch.Tensor:
    T = max(float(T), 1e-6)
    z = sims / T
    z = z - z.max()
    p = torch.exp(z)
    return p / (p.sum() + 1e-12)
```

### 4.5 Memory Construction

```python
@torch.no_grad()
def build_system_complex(
    D, vocab_words, sentences, L, ensembles, banks,
    device, dtype, orth_vocab_iters=1, superpose_scale=True
):
    E = []
    V, S = len(vocab_words), len(sentences)
    idx_w = {w: i for i, w in enumerate(vocab_words)}

    items = [(sid, pos, w) for sid, sent in enumerate(sentences)
             for pos, w in enumerate(sent)]
    items_by_bank = [[] for _ in range(banks)]
    for k, it in enumerate(items):
        items_by_bank[k % banks].append(it)

    for _ in range(ensembles):
        banks_list = []
        for b in range(banks):
            binder = ComplexBinder(D, device, dtype)
            vocab = make_vocab_keys_complex(V, D, device, dtype)
            if orth_vocab_iters > 0:
                vocab = orthogonalize_complex(vocab, iters=orth_vocab_iters)
            sent_keys = make_sentence_keys_complex(S, D, device, dtype)
            role_keys = make_role_keys_power(D, L, device, dtype)
            bindings = []
            for (sid, pos, w) in items_by_bank[b]:
                Skey = sent_keys[sid]
                Rkey = role_keys[pos]
                Wkey = vocab[idx_w[w]]
                bindings.append(binder.bind(binder.bind(Skey, Rkey), Wkey))
            if len(bindings) == 0:
                memory = torch.zeros(D, device=device,
                    dtype=torch.complex64 if dtype==torch.float32 else torch.complex128)
            else:
                memory = torch.stack(bindings).sum(dim=0)
                if superpose_scale:
                    memory = memory * (1.0 / math.sqrt(len(bindings)))
            banks_list.append(dict(
                binder=binder, memory=memory, vocab=vocab,
                sent=sent_keys, roles=role_keys
            ))
        E.append(dict(banks=banks_list))
    return E
```

### 4.6 Decoding / Retrieval

```python
@torch.no_grad()
def decode_position(E_data, sid, pos, vocab_words,
                    sharpen_p, corvacs_a, temp_T, topk):
    V = len(vocab_words)
    device = E_data[0]['banks'][0]['vocab'].device
    sims_sum = torch.zeros(V, device=device, dtype=torch.float32)

    for ens in E_data:
        for bank in ens['banks']:
            b = bank['binder']
            rec = b.unbind(b.unbind(bank['memory'], bank['sent'][sid]),
                           bank['roles'][pos])
            rec = corvacs_lite(sharpen_complex(rec, p=sharpen_p), a=corvacs_a)
            rec2 = torch.cat([rec.real, rec.imag], dim=-1)
            rec2 = rec2 / (rec2.norm() + 1e-12)
            sims = bank['vocab_norm'] @ rec2
            sims_sum += sims

    probs = softmax_temp(sims_sum, T=temp_T)
    best_idx = int(torch.argmax(probs))
    return vocab_words[best_idx], sims_sum, probs
```

### 4.7 Upsert & Rebuild

```python
def upsert_fact_fast(tag, value_text, evidence="", page=None):
    if not tag or not value_text: return
    found = False
    for i, f in enumerate(facts):
        if f["key"].lower() == tag.lower():
            facts[i]["value_text"] = value_text
            facts[i]["evidence"] = evidence or facts[i].get("evidence","")
            facts[i]["page"] = page if page is not None else facts[i].get("page", None)
            found = True; break
    if not found:
        facts.append({"key": tag, "value_text": value_text,
                       "evidence": evidence, "page": page})
    gold[tag.lower()] = value_text
    global E, vocab_words, tag_to_pos
    E, vocab_words, tag_to_pos = rebuild_memory_fast(facts)

def rebuild_memory_fast(facts_list):
    uniq_vals, seen = [], set()
    for f in facts_list:
        v = f["value_text"]
        if v not in seen:
            uniq_vals.append(v); seen.add(v)
    L = max(1, len(facts_list))
    sentences = [[facts_list[i]["value_text"] for i in range(L)]]
    tag_to_pos = {facts_list[i]["key"]: i for i in range(L)}
    E = build_system_complex(
        D=FAST_D, vocab_words=uniq_vals, sentences=sentences, L=L,
        ensembles=1, banks=FAST_BANKS, device=device, dtype=rdtype,
        orth_vocab_iters=1, superpose_scale=True
    )
    return E, uniq_vals, tag_to_pos
```

### 4.8 Alias Resolution

```python
def best_tag(user_text, tags):
    text = user_text.lower()
    for tag, al in ALIASES.items():
        for a in al + [tag.lower()]:
            if a in text and tag in tags:
                return tag
    best, best_s = "", 0.0
    for tag in tags:
        s = SequenceMatcher(None, text, tag.lower()).ratio()
        if s > best_s:
            best, best_s = tag, s
    return best if best_s >= 0.55 else ""
```

### 4.10 Metrics

```python
def fuzzy_score(a, b):
    return SequenceMatcher(None, a.lower().strip(), b.lower().strip()).ratio()

def summarize_metrics(tag, answer_text, probs, sims, src):
    conf = margin = None
    if probs is not None:
        topv, topi = torch.topk(probs, k=2)
        conf = float(topv[0])
        if sims is not None:
            s_top2 = torch.topk(sims, k=2).values
            margin = float((s_top2[0] - s_top2[1]).item())
        else:
            margin = float(topv[0] - topv[1])
    g = gold.get((tag or "").lower())
    if g:
        stats["total"] += 1
        if fuzzy_score(answer_text, g) >= 0.72:
            stats["correct"] += 1
    suffix = [f"acc {stats['correct']}/{stats['total']}"]
    if conf is not None:   suffix.append(f"conf {conf:.3f}")
    if margin is not None: suffix.append(f"margin {margin:.3f}")
    suffix.append(f"src {src}")
    return " [" + " | ".join(suffix) + "]"
```

## 6. The Hybrid Query Pipeline

The system resolves queries in a strict priority order:

1. **Alias matching** (`best_tag`): Substring match against known aliases, then fuzzy SequenceMatcher fallback (threshold ≥ 0.55). Maps natural language to a stored tag.

2. **HRR decode** (`decode_position`): If tag found, unbind from memory. If result is substantive (>1 word, not just "yes"/"no"), return it. Source: `HRR`.

3. **LLM fallback** (`pdf_qa_answer_cached`): If HRR answer is weak, query Gemini with the full PDF. Cache the response. Source: `PDF`.

4. **Upsert** (`upsert_fact_fast`): The LLM's answer is added to the fact store and the entire HRR memory is rebuilt. Future queries for the same tag will hit HRR, not the LLM.

**Key property**: The system is **self-improving**. Every LLM fallback makes future HRR retrievals more likely to succeed. The LLM call rate decreases over time.

## 7. Signal Processing Chain

After unbinding, the raw retrieved vector is noisy. Three operations clean it up:

### 7.1 Sharpening (`sharpen_complex`)

```
z_sharp = z × |z|^(p-1)
```

- `p = 1.0`: No effect (identity)
- `p = 1.25`: Amplifies high-magnitude components (signal), suppresses low-magnitude components (noise)
- `p = 0.8`: Softens — reduces contrast

**Effect**: Increases signal-to-noise ratio by making the magnitude distribution more peaked.

### 7.2 Magnitude Limiting (`corvacs_lite`)

```
z_limited = z × tanh(a × |z|) / |z|
```

- `a = 0`: Disabled
- `a = 0.9`: Moderate saturation — large magnitudes compressed, small pass through
- `a = 2.0`: Strong saturation — almost binary magnitude

**Effect**: Prevents any single dimension from dominating the cosine similarity. Acts as automatic gain control.

### 7.3 Temperature Softmax

```
p_i = exp(sim_i / T) / Σ_j exp(sim_j / T)
```

- `T = 0.7`: Sharpened distribution — confident top pick
- `T = 1.0`: Standard softmax
- `T = 2.0`: Flattened distribution — less confident

### Combined Pipeline

```
raw → sharpen(p=1.25) → corvacs(a=0.9) → cosine_sim → softmax(T=0.7) → argmax
```

This is a **denoising pipeline** for holographic retrieval. It's analogous to how a radio receiver has an amplifier (sharpen), a limiter (corvacs), and a tuner (softmax).

## 8. Memory Lifecycle

```
1. EXTRACT:  PDF → Gemini → structured facts [{key, value_text, page, evidence}]
2. BUILD:    facts → vocabulary + sentence/role keys → bind triples → superpose → memory tensor
3. QUERY:    user question → tag resolution → unbind from memory → decode → answer
4. LEARN:    weak answer → LLM fallback → upsert new fact → FULL REBUILD of memory
5. REPEAT:   future queries for same tag → HRR hit (no LLM call)
```

**Important**: Upsert triggers a FULL REBUILD of the memory tensor. This is expensive but ensures consistency. There is no incremental update — the entire superposition is reconstructed.

## 9. Hyperparameters & Tuning

| Parameter | Fast Config | Full Config | Effect |
|-----------|:-----------:|:-----------:|--------|
| `D` (dimension) | 1024 | 4096-8192 | Higher = more capacity, less interference |
| `BANKS` | 4 | 12 | More banks = fewer items per bank = cleaner retrieval |
| `ENSEMBLES` | 1 | 2 | More ensembles = variance reduction via averaging |
| `SHARPEN_P` | 1.0 | 1.25-1.3 | Higher = more aggressive noise suppression |
| `CORVACS_A` | 0.0 | 0.9-1.1 | Higher = stronger magnitude limiting |
| `TEMP_T` | 0.9 | 0.65-0.7 | Lower = sharper probability distribution |
| `orth_vocab_iters` | 1 | 1 | More iters = more orthogonal vocab (diminishing returns) |
| `orth_step` | 0.4 | 0.4 | Learning rate for orthogonalization |
| `fuzzy_threshold` | 0.72 | 0.72 | Min similarity to gold for "correct" |
| `alias_threshold` | 0.55 | 0.55-0.60 | Min SequenceMatcher ratio for tag matching |

### Capacity Rule of Thumb

Reliable retrieval capacity per bank ≈ `√D`:
- D=1024 → ~32 items/bank
- D=4096 → ~64 items/bank
- D=8192 → ~90 items/bank

Total system capacity ≈ `BANKS × √D`:
- 4 banks × √1024 ≈ 128 facts
- 12 banks × √4096 ≈ 768 facts

## 10. Capacity, Interference & Limits

### What happens as facts increase:

| Facts | Expected Accuracy | Confidence | Notes |
|:-----:|:-----------------:|:----------:|-------|
| 10 | ~99% | >0.1 | Clean retrieval |
| 50 | ~95% | ~0.05 | Minor interference |
| 100 | ~90% | ~0.01 | Noticeable noise |
| 250 | ~88% | ~0.006 | Significant interference |
| 500+ | <80% | <0.003 | Unreliable without more banks/dimensions |

### Mitigation strategies in the code:
1. **Sharding** (banks): Distributes facts round-robin, reduces per-bank load
2. **Ensembles**: Multiple independent copies, averaged for noise reduction
3. **Orthogonal vocab**: Reduces cross-word interference
4. **Signal processing**: Sharpen + corvacs + temperature softmax
5. **Superpose scaling**: `1/√N` normalization prevents magnitude explosion

### What the system does NOT handle:
- No forgetting mechanism (all facts persist equally)
- No priority/importance weighting
- No incremental update (full rebuild on every upsert)
- No dynamic capacity expansion

OpenClaw currently uses markdown files + SQLite (hybrid BM25 + vector search) for memory. HRR can augment this as a **fast working memory layer**.

```
│                   │                               │
│          Query Resolution Order:                  │
│          1. HRR (microseconds)                    │
│          2. SQLite search (milliseconds)          │
│          3. File read (if specific path)          │
│                                                   │
│          Learning:                                │
│          - Agent writes to markdown (durable)     │
│          - System extracts facts → HRR (fast)     │
│          - Pre-compaction: flush HRR → markdown   │
└───────────────────────────────────────────────────┘
```

### What to build:

1. **HRR Memory Manager** — A service that maintains a complex tensor in memory (D=4096, 4-8 banks). Exposes:
   - `remember(key: str, value: str)` — binds fact into memory (with batch rebuild)
   - `recall(query: str) -> (answer, confidence, margin)` — unbinds and decodes
   - `forget(key: str)` — removes fact and rebuilds
   - `save() / load()` — serialize tensor to disk

2. **Integration with OpenClaw's `memory_search`** — Before the existing hybrid BM25+vector search, check HRR first. If HRR returns high-confidence answer (conf > threshold), skip the SQLite search entirely.

3. **Learning hook** — When OpenClaw's agent writes a new fact to markdown (via its `memory` tool), extract the key-value pair and upsert into HRR.

4. **Pre-compaction hook** — When approaching context limits, dump HRR contents to a structured markdown file for durability. On next boot, reload from that file.

### Key benefits for OpenClaw:
- **Faster repeated queries**: HRR is O(1), no database query needed
- **Structured memory**: Key-value facts, not text chunks
- **Session continuity**: Tensor can be saved/loaded between sessions
- **Reduced embedding costs**: No need to embed frequently-accessed facts

## 12. How to Adapt for Claude Code

Claude Code uses `MEMORY.md` files and project-level `CLAUDE.md` for persistent context. HRR can serve as a **compressed project knowledge cache**.

### Integration Architecture

```

│                  Claude Code Agent                 │
│                                                    │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────┐ │
│  │ HRR Project  │  │  Grep/Glob   │  │ CLAUDE.md│ │
│  │ Memory       │  │  (codebase   │  │ MEMORY.md│ │
│  │ (learned     │  │   search)    │  │ (static  │ │
│  │  patterns)   │  │              │  │  config) │ │
│  └──────┬───────┘  └──────┬───────┘  └────┬─────┘ │
│         │                 │               │       │
│  Use cases for HRR:                               │
│  1. "Where is X defined?" → HRR if learned before │
│  2. "What pattern does this project use?" → HRR   │
│  3. "What was the fix for that bug?" → HRR        │
│  4. File path associations → HRR                  │
│                                                    │
│  Learning triggers:                                │
│  - After successful Grep/Glob → cache result      │
│  - After reading a file → extract key facts        │
│  - After fixing a bug → store diagnosis + fix      │
│  - User says "remember this" → direct upsert      │
└────────────────────────────────────────────────────┘
```

### What to build:

1. **Project-scoped HRR tensors** — One tensor per project, stored in `.claude/projects/<hash>/hrr_memory.pt`. Contains learned facts like:
   - `"test command" → "pytest src/tests -v"`
   - `"auth module" → "src/auth/middleware.ts"`
   - `"database ORM" → "Prisma with PostgreSQL"`
   - `"deploy process" → "git push to main triggers Vercel"`

2. **Auto-learning from tool results** — When Claude Code successfully finds something via Grep/Glob, cache the mapping:
   - Query: "where is the login handler"
   - Result: `src/routes/auth.ts:47`

3. **Compressed context injection** — Instead of loading the full MEMORY.md into the system prompt, encode its facts into HRR and query on-demand. This saves context window space.

4. **Cross-session pattern learning** — Track which files are frequently read together, which functions are often modified together, which error patterns recur. Encode these as HRR facts.

### Key benefits for Claude Code:
- **Context window savings**: Facts in HRR don't consume prompt tokens
- **Faster project orientation**: Instant recall of project patterns without re-reading files
- **Accumulated intelligence**: Gets smarter about the project over sessions
- **No infrastructure**: Just a `.pt` file, no database needed

## 13. How to Adapt for Codex / OpenAI Agents

OpenAI's Codex (and agent frameworks like Swarm, Assistants API) use tool calls and retrieval. HRR can serve as a **lightweight agent memory** that avoids the Assistants API's file search overhead.

### Integration Architecture

```
┌────────────────────────────────────────────────────┐
│              Codex / OpenAI Agent                  │
│                                                    │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────┐ │
│  │ HRR Working  │  │  File Search │  │ Function │ │
│  │ Memory       │  │  (Assistants │  │ Calling  │ │
│  │ (per-thread) │  │   API)       │  │          │ │
│  └──────┬───────┘  └──────┬───────┘  └────┬─────┘ │
│         │                 │               │       │
│  Exposed as a tool:                               │
│  {                                                │
│    "name": "hrr_memory",                          │
│    "description": "Fast key-value memory...",     │
│    "parameters": {                                │
│      "action": "recall|remember|forget",          │
│      "key": "string",                             │
│      "value": "string (for remember)"             │
│    }                                              │
│  }                                                │
│                                                    │
│  The agent calls hrr_memory.recall("X") before    │
│  falling back to file_search or web browsing.     │
└────────────────────────────────────────────────────┘
```

### What to build:

1. **HRR as a tool** — Expose `recall`, `remember`, `forget` as function calls. The agent learns to check memory before making expensive retrieval calls.

2. **Thread-scoped memory** — Each conversation thread gets its own HRR tensor. Facts accumulated in one thread persist for the thread's lifetime.

3. **Cross-thread knowledge distillation** — Periodically extract high-confidence facts from thread-level HRR memories and merge into a global HRR tensor.

4. **Confidence-gated retrieval** — The tool returns confidence alongside the answer. The agent can decide whether to trust the HRR answer or fall back to file search.

### Key benefits for Codex/OpenAI:
- **Reduced API costs**: Fewer file search calls (billed per call)
- **Lower latency**: HRR recall is instant vs. file search latency
- **Agent autonomy**: The agent manages its own memory via tool calls
- **No vector store needed**: Eliminates the Assistants API vector store overhead

```python
        tag = self._resolve_tag(query)
        if tag not in self.tag_to_pos:
            return {"answer": None, "confidence": 0, "margin": 0, "found": False}
        answer, sims, probs = self._decode(tag)
        conf = float(torch.topk(probs, 2).values[0])
        margin = float(torch.topk(sims, 2).values[0] - torch.topk(sims, 2).values[1])
        return {"answer": answer, "confidence": conf, "margin": margin, "found": True}

        self.facts = [f for f in self.facts if f["key"].lower() != key.lower()]
```

```python
def agent_answer(query: str, memory: HRRMemory, llm_fallback) -> str:
    # 1. Check HRR memory first (microseconds, free)
    result = memory.recall(query)
    if result["found"] and result["confidence"] > CONFIDENCE_THRESHOLD:
```

## 15. What Makes This Different from RAG

| Dimension | RAG | HRR Memory |
|-----------|-----|------------|
| **Mental model** | Filing cabinet | Holographic sticky notes |
| **Storage** | Separate vectors in database | All facts in one superposed tensor |
| **Retrieval** | "Find text similar to query" | "Extract value bound to this key" |
| **Returns** | Text chunks (exact) | Decoded value (noisy, needs cleanup) |
| **Infrastructure** | Embedding model + vector DB + chunking | One PyTorch tensor (~32KB) |
| **Capacity** | Millions of documents | ~√D × banks facts (~128-768) |
| **Latency** | 10-100ms (DB query + embedding) | <1ms (tensor multiply) |
| **Cost per query** | Embedding API call | Zero (local compute) |
| **Compositionality** | None (LLM reasons over text) | Native (role-filler bindings) |
| **Learning** | Re-index documents | Upsert fact + rebuild tensor |
| **Best for** | Large knowledge bases | Small, hot, frequently-accessed facts |

### They are complementary, not competing:

```
HRR = L1 cache (tiny, fast, lossy, associative)
RAG = L2 cache (large, slower, exact, indexed)
LLM = Main memory (huge, slowest, generative)
```

The notebook demonstrates exactly this layered pattern: HRR first, then LLM (PDF QA) fallback, with upsert to promote LLM answers into HRR.

## 16. Known Limitations & Open Problems

### Current limitations:
1. **Full rebuild on every upsert** — O(N × D) to add one fact. Need incremental superposition update.
2. **No forgetting/decay** — All facts have equal weight. Need importance-weighted superposition.
3. **Flat key namespace** — Tags are strings, no hierarchy. Need compositional keys for nested knowledge.
4. **Alias system is static** — Predefined alias maps, not learned. Need LLM-generated or embedding-based tag resolution.
5. **No confidence calibration** — Raw softmax probabilities are not calibrated. Need proper thresholding.
6. **Vocabulary is closed** — Can only retrieve values that were in the vocabulary at build time. New values require rebuild.

### Open problems for production use:
1. **Incremental update**: Can we add a new binding to the superposition without rebuilding? (Yes, theoretically: just add the new binding vector. But role keys and vocab keys must be preserved across updates.)
2. **Graceful capacity expansion**: Can we grow D or banks dynamically when interference gets too high?
3. **Hierarchical HRR**: Can we nest HRR memories (e.g., topic-level memory → fact-level memory) to scale beyond √D?
4. **Learned encoders**: Can we train a neural network to map natural language to HRR keys, replacing the manual alias system?
5. **Differentiable integration**: Can HRR be a differentiable layer in a transformer, as Hrrformer suggests?
